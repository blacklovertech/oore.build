# Setup Wizard

The setup wizard is the first-run configuration flow that takes a fresh oore.build instance from `bootstrap_pending` to `ready` state. It is available both as a [web UI](/features/setup-wizard-ui) and as an interactive [CLI flow](/cli/setup).

## Overview

Every oore.build instance must complete a 4-step setup process before it can accept authenticated users:

1. **Bootstrap Token** -- Verify the one-time token generated by `oore setup open`
2. **OIDC Configuration** -- Configure the identity provider (issuer URL, client ID, optional client secret)
3. **Owner Account** -- Verify the platform owner's identity via OIDC
4. **Complete** -- Finalize setup and permanently lock all setup endpoints

::: tip
Setup can be driven from the web UI at `/setup` or interactively from the terminal with `oore setup`. Both use the same backend API.
:::

## State Machine

The setup state machine enforces a strict progression:

```
uninitialized --> bootstrap_pending --> idp_configured --> owner_created --> ready
```

| State | Description |
|---|---|
| `uninitialized` | Initial state before any setup action |
| `bootstrap_pending` | Instance created, waiting for bootstrap token verification |
| `idp_configured` | OIDC identity provider has been configured |
| `owner_created` | Owner account has been verified via OIDC |
| `ready` | Setup complete, all setup endpoints permanently disabled |

::: warning
State transitions are one-way. Once setup reaches `ready`, all setup endpoints return `409 Conflict` and cannot be re-enabled without direct database intervention.
:::

## Step 1: Bootstrap Token

The operator generates a time-bound bootstrap token on the host machine:

```bash
oore setup open --ttl 15m
```

This generates a 32-byte random token (via `OsRng`), stores only its SHA-256 hash in the database, and prints the plaintext to the terminal. The token is:

- **One-time use** -- consumed on first verification, rejected if reused
- **Time-bound** -- expires after the specified TTL (default: 15 minutes)
- **Hash-only storage** -- the plaintext is never persisted

The user then submits this token to `POST /v1/setup/bootstrap-token/verify`. On success, the token is marked consumed and the server returns a **setup session token** (30-minute sliding-window TTL).

## Step 2: OIDC Configuration

With the setup session token, the operator configures their OIDC identity provider by submitting the issuer URL, client ID, and optional client secret to `POST /v1/setup/oidc/configure`.

The daemon:

1. Performs [OIDC discovery](https://openid.net/specs/openid-connect-discovery-1_0.html) on the issuer URL (fetching `.well-known/openid-configuration`)
2. Validates that required endpoints exist (authorization, token, JWKS)
3. If a client secret is provided, encrypts it with [AES-256-GCM](/security/overview#encryption-at-rest) before storing
4. Transitions the state to `idp_configured`

::: danger
This step cannot be undone. Double-check your OIDC provider values before submitting.
:::

## Step 3: Owner Verification

The owner account is created by completing an OIDC authorization code flow with [PKCE](/security/overview#pkce-flow). This proves that the operator controls the email address registered with the identity provider.

The flow works as follows:

1. The client calls `POST /v1/setup/owner/start-oidc` with a `redirect_uri`
2. The daemon returns an `authorization_url` for the IdP
3. The user authenticates with their IdP and is redirected back with a `code` and `state`
4. The client submits these to `POST /v1/setup/owner/verify-oidc`
5. The daemon exchanges the code for tokens, verifies the ID token, and extracts the `email` and OIDC `subject`
6. The owner is created and the state transitions to `owner_created`

## Step 4: Complete

The final step calls `POST /v1/setup/complete`, which:

- Transitions the state to `ready`
- Clears the setup session (the session token becomes invalid)
- Returns the `instance_id`

After completion, all setup endpoints are permanently disabled and the [auth endpoints](/api/auth) become available.

## Security Considerations

- Bootstrap tokens use 256 bits of cryptographic randomness
- Only SHA-256 hashes of tokens are stored; plaintext is never persisted
- Setup session uses a 30-minute sliding-window TTL (renewed on each authenticated request)
- OIDC client secrets are encrypted with AES-256-GCM at rest
- The public `/v1/public/setup-status` endpoint exposes only non-sensitive state information
- All setup endpoints are permanently disabled after reaching `ready` state

See the [Security Overview](/security/overview) for more details.
